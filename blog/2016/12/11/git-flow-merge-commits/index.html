Git flow中的Merge Commits

发表于 2016-12-11   |   分类于 工具集   |     |   阅读次数

Git flow )是一个成熟的Git branching model，在实际的开发流程中，为保持代码同步，或者需要另一条分支的代码，经常会做分支Merge的操作，产生了大量的无意义的commits，如下图。

这样不仅会让git log的路径变曲折，而且也给通过commit history定位问题增加了难度。
本文来谈一谈如何避免这类提交。

git stash

多人在一个分支上协同开发时，一个好的习惯是开发前先执行git pull操作，保持远程的代码和本地的代码同步，但即使这样也无法避免在开发的过程中，他人产生了commits，并push到了远程。

可以在commit之前，准确地说，是在git add任何东西之前，先通过git stash save把变更暂存起来， 然后执行git pull来同步远程更新，最后通过git stash pop命令暂存的改动应用到已经与远程同步过的代码base上，再add、commit、push。

git cherry-pick

还是基于git rebase所假设的场景，另一种把develop分支的代码改动应用到自己的feature分支的手段就是cherry-pick。故名思议，该命令就是把某一个或某几个commits，从其它分支pick过来。这要求其它分支的commits之间关联度比较小，commit的粒度刚好合适。否则，你cherry-pick了一个commit，结果这个commit的变更是基于它的前一个commit，这么拔出萝卜带出泥可就不好玩了。

shell ➜ ae git:(feature/awesome-feature) git cherry-pick 7075fcd shell
